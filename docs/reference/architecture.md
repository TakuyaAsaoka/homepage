# アーキテクチャと設計思想

このサイトの技術的な設計判断を、後から「なぜこの構成なのか（WHY）」を思い出せるようにまとめたリファレンスです。将来の保守者（＝未来の自分）向け。

---

## 1. 概要

このサイトは、次の3つの柱でできています。

- **静的サイト生成** — ビルド時にHTMLを作り切る。実行時のサーバもデータベースも要らない。
- **唯一の正（Single Source of Truth）はgit** — すべてのコンテンツ・設定はリポジトリ内のファイル。編集の入口が何であれ、行き着く先はここ。
- **役割ごとにデータ形式を選ぶ** — 本文はMarkdown、設定はYAML、機械連携はJSON。

全体像:

```
                     ┌─────────────────────┐
  [入口1] CMS ──────►│                     │
   (/admin)          │   Git リポジトリ     │      ┌─────────────┐     ┌──────────────┐
                     │  （唯一の正・SoT）   ├─push─►│ GitHub Actions├────►│ GitHub Pages │
  [入口2] 直接編集 ──►│  src/ 配下のファイル  │      │  ビルド        │     │   公開サイト   │
   (エディタ)         │                     │      └─────────────┘     └──────────────┘
                     └─────────────────────┘
```

編集の入口は2つあるが、どちらも「リポジトリのファイルを書き換えてコミットする」ことに収束する。だから正本は常に1つで、食い違いが起きない。

---

## 2. Astroの特徴と採用理由

Astroは静的サイト生成フレームワーク。このサイトが受けている具体的な恩恵は次のとおり。

| 特徴 | 中身 | このサイトでの恩恵 |
|------|------|------------------|
| **静的サイト生成（SSG）** | ビルド時にHTMLを生成し、公開物はただのHTML/CSS/JS | 実行時のサーバ・DBが不要。高速・堅牢で、GitHub Pagesに無料で置ける |
| **コンテンツコレクション + zod** | `src/content/` 配下のファイルを、`content.config.ts` のスキーマで型検証 | 不正な値（形式違反・必須欠落）を**ビルド時に検出**。壊れたまま公開されない |
| **基本ゼロJS** | 必要な箇所だけJavaScriptを送る | ページが軽く、表示が速い |
| **ファイルベースルーティング** | `src/pages/` のファイル配置がそのままURLになる | 例: `src/pages/works/index.astro` → `/works/`。URL構造がディレクトリで一目瞭然 |

要するに「**書いた内容がビルド時に検証され、静的なHTMLとして焼き上がる**」ため、公開後に動く部品が少なく、壊れにくい。

---

## 3. 編集からデプロイまでの仕組み

編集内容が公開されるまでの流れ。ポイントは、**入口が違っても最終的にすべて `main` へのコミットに集約される**こと。

```
[入口1] CMS で保存 ──┐
                     ├──► main ブランチに commit
[入口2] 直接編集push ─┘            │
                                  ▼
              GitHub Actions（.github/workflows/deploy.yml）
                  ├─ check : 型チェック（npm run check）
                  ├─ build : Astro が HTML を生成
                  └─ deploy: GitHub Pages へ公開
                                  │
                                  ▼
                        数分後、公開サイトに反映
```

- **CMSもデータベースを持たない。** Sveltia CMSは管理画面から**同じリポジトリのファイルを書き換えてコミットするGUI**にすぎない。裏に別の保管場所はない。
- **静的なので実行時サーバが不要。** 公開されるのは焼き上がったHTML等の静的ファイルだけ。攻撃面が小さく、落ちにくく、安い。
- **プルリクエストでは `check`（型チェック）のみ走り、デプロイは `main` への push 時**に行われる（`build` は `pull_request` を除外）。

## 4. データの持ち方を役割で決める

コンテンツ・設定は、その「形」に合わせて形式を使い分けている。行き当たりばったりではなく、次の基準による。

> **自由な本文があるもの → Markdown / 決まった項目の集まり → YAML / 機械が読み書きするもの → JSON**

| 形式 | 得意なこと | このサイトでの用途 | 実ファイル |
|------|-----------|------------------|-----------|
| **Markdown (.md)** | 長い自由な本文（見出し・箇条書き・リンク）＋フロントマター（構造化データ） | 制作（記事的で、本文がある。今後増える集まり） | `src/content/works/*.md` |
| **YAML (.yaml)** | 決まった項目の構造化データ。コメント可・複数行が書きやすい・記号が少ない | Home・Aboutの文言、サイト設定（項目を埋めるだけで本文がない固定ページ） | `src/content/pages/home.yaml`, `about.yaml`, `src/content/settings/site.yaml` |
| **JSON (.json)** | 機械が読み書きするデータ。厳密で曖昧さがない | 現状は未使用。将来、外部システムとの自動連携や翻訳ファイル等が出てきたときの選択肢 | — |

補足: YAMLはJSONのほぼ上位互換で、人が手で書くのに向く（コメントが書ける・引用符やカンマが少ない）。逆にJSONはコメントを一切書けず手編集はつらいが、プログラム間のデータ交換では共通語。**このサイトの設定は人（とCMS）が手で編集する**ため、YAMLが最適だった。

例（Home の YAML は「項目を埋める」形）:

```yaml
hero:
  role: SOFTWARE ENGINEER / SRE   # 肩書き
  tagline: つくることと、動かし続けること。  # キャッチコピー
skills:
  - TypeScript
  - React
```

## 5. どこを触ればどこが変わるか

保守時の早見表。「変えたいもの」から「編集する場所」と「CMSの入口」を引ける。

| 変えたいもの | 編集する場所（ファイル） | CMSの入口 |
|-------------|----------------------|-----------|
| 制作の追加・編集 | `src/content/works/*.md` | 制作 |
| Homeの文言（肩書き・キャッチコピー・紹介文・スキル） | `src/content/pages/home.yaml` | ページ → Home |
| Aboutの文言・セクション | `src/content/pages/about.yaml` | ページ → About |
| サイト名・SNSリンク・メール・note RSS等 | `src/content/settings/site.yaml` | サイト設定 |
| ページの構造・デザイン | `src/pages/*.astro`, `src/components/*`, `src/styles/global.css` | （コード編集のみ） |
| 技術的な定数（`BASE_PATH`・ロケール等） | `src/consts.ts` | （コード編集のみ） |

「文言・設定」はCMSからも編集できるが、「構造・デザイン・技術定数」はコードを直接編集する。この線引きが、非エンジニアの編集範囲とエンジニアの責務を分けている。

---

## 6. 設計原則（まとめ）

- **唯一の正はgit** — データの居場所を1つに集約。CMSも直接編集も、同じリポジトリのファイルに書き込む。だから食い違わない。
- **役割に合った形式** — 本文はMarkdown、設定はYAML、機械連携はJSON。適材適所でデータを持つ。
- **型安全** — zodスキーマで不正な値をビルド時に検出し、壊れたまま公開しない。
- **静的で堅牢・低コスト** — サーバレス。公開物は静的ファイルだけで、GitHub Pagesに無料で置ける。
- **非エンジニアも編集できる** — CMSがMarkdown/YAMLの存在を隠し、項目を埋めるだけで更新できるようにしている。
